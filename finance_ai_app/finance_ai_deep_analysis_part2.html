Finance AI — data flows, code quality audit, and refactoring roadmap

End-to-end data flows

Flow A — add a transaction (happy path)

1
User opens TransactionDialog (Ctrl+N or button)
ShortcutManager or toolbar button calls TransactionDialog(parent). Form pre-fills today's date; category defaults to "Auto (AI)".
↓
2
Dialog accepted → TransactionManager.add_transaction()
Facade validates, delegates to TransactionRepo.add(model). Model.validate() checks amount > 0, date format, description non-empty before any SQL.
↓
3
DB write inside db_session() context manager
INSERT into transactions; UPDATE accounts.balance ± amount in the same atomic context. Rolls back both on any exception.
↓
4
Budget alert check
_check_budget_alert() queries real spent vs limit for this category+month. If ratio ≥ alert_threshold → bus.notify_warning.emit().
↓
5
bus.transaction_added.emit() + bus.balance_changed.emit()
All subscribed frames (Dashboard, TransactionFrame, BudgetFrame) receive signals and call their own refresh(). Sidebar also refreshes balance label.
↓
6
Auto-classify worker (background QThread)
If auto_classification=True in settings, ClassifyWorker runs TransactionClassifier.predict_category_id(description) and updates the row. On finish, emits another refresh.

Flow B — AI forecast run

1
User clicks "Chạy dự báo AI" in ForecastFrame
Button disabled; progress bar shown; ForecastWorker(QThread) spawned.
↓
2
SpendingForecaster.forecast_all_categories()
Iterates every expense category. For each: loads monthly totals as pandas DataFrame, selects Prophet vs moving-average based on data length and settings.
↓
3
Results written back to ai_predictions table
DELETE old predictions for next month, then INSERT new rows with predicted_amount + confidence. Next time Dashboard loads, it reads from this table for the "AI forecast" KPI card.
↓
4
Worker emits finished(results) → UI updates
ForecastFrame renders category bar chart, updates KPI cards, calls get_forecast_chart_data() for the line chart with confidence band.

Flow C — CSV bank import

1
File picker → CsvImporter.import_file(path, bank="auto")
Bank auto-detected from filename keywords first, then first 500 chars of file content.
↓
2
Multi-encoding read → fuzzy column mapping
Tries utf-8-sig → utf-8 → latin-1. Maps columns by substring match against the bank profile's candidate list. Normalises dates (up to 3 format attempts) and amounts (strips thousands separators, detects comma-vs-dot ambiguity).
↓
3
Atomic batch insert with rollback guard
All rows inserted in a single transaction. Any failure rolls back the entire import — no partial data. Account balances updated per-row inside the same transaction.
↓
4
Count + warnings returned → refresh triggered
UI shows a summary dialog. bus.transactions_imported.emit(count) signals all frames to refresh.

Code quality audit

Issues ranked by severity: high affects correctness or crashes, medium causes maintenance pain, low is style/minor.

high
Two DB access patterns coexist — connection lifecycle conflict
models.py (singleton conn) vs repositories.py (per-call conn)
DatabaseManager keeps a single check_same_thread=False shared connection and commits in __exit__. Meanwhile db_session() opens fresh connections. Background QThreads using the singleton path can see uncommitted data from another thread or deadlock under concurrent writes. The execute_write() lock only serialises one path.
high
transaction_manager.py imports from wrong module path
transaction_manager.py line 3–4
from app.repositories import TransactionRepo but the actual file is repositories.py; similarly from app.repositories import TransactionModel. This will raise ModuleNotFoundError at runtime unless __init__.py re-exports it. If the flat layout is intentional, import paths should be from repositories import ....
high
goal_tracker.py uses raw get_connection() as if it's sqlite3.Connection
goal_tracker.py — GoalTracker.__init__ and methods
self.conn = get_connection() returns a DatabaseManager proxy, not a real connection. Then calls self.conn.execute_write() which exists on DatabaseManager, but also calls self.conn.execute() directly without committing. The connection is stored as an instance attribute and never closed — resource leak.
medium
settings_frame.py calls theme_engine.mode() as a method — it's a property
settings_frame.py — _initial_refresh(), refresh()
ThemeEngine.mode is a @property in theme_engine.py. Calling theme_engine.mode() with parentheses will raise TypeError: 'str' object is not callable. Should be theme_engine.mode.
medium
AnomalyDetector opens/closes raw connections without context manager
anomaly_detector.py — detect_and_mark(), get_anomalies(), explain_anomaly()
Multiple methods call conn = get_connection() then conn.close() manually. If an exception is raised mid-method, close() is never called. Should use with db_session() as db or with get_connection() as conn.
medium
Hard-coded account_id=1 in several places
transaction_frame.py (TransactionDialog.get_data), csv_importer.py, command_palette.py
Multi-account support exists in the schema but account_id is silently forced to 1 in the dialog output and CSV import. Adding a second account will not reflect in any imported or command-palette-added transaction.
medium
forecast_frame.py stores QThread workers as instance attributes without cleanup
forecast_frame.py — _run_forecast(), _run_anomaly()
self.fc_worker and self.anom_worker are overwritten on each run without waiting for the previous thread. If the user clicks "Run" twice quickly, the first worker's finished signal may fire after the second is already running — results interleave.
low
Inline stylesheet strings duplicated across every frame
All *_frame.py files — static _btn_style(), _combo_style() methods
Each frame defines its own _btn_primary(), _btn_normal(), _combo_style() static methods returning identical QSS strings. These should be moved to a shared ui_styles.py module or handled entirely by ThemeEngine.
low
error_handler.py uses file mtime as timestamp — always wrong
error_handler.py line ~14
f"Timestamp: {Path('data/error.log').stat().st_mtime}\n" reads the file's modification time (a float) rather than the current datetime. Should be datetime.now().isoformat().
note
Family/group features are UI stubs — not connected to DB
family_frame.py
The DB schema has family_groups and group_members tables, but FamilyFrame only calls QInputDialog and shows a random invite code. No DB writes or reads are performed. user_switched bus signal has no consumers.

Dependency risk

Four packages are optional-but-silently-required for core advertised features: prophet (forecast), sklearn (classifier + anomaly), reportlab (PDF), torch + transformers (embedded LLM). None of these have install-time validation — the app starts fine and only errors when the user clicks the feature. A startup dependency check with a guided install dialog would improve UX significantly.

Refactoring roadmap

Prioritised by impact vs effort. Each item is self-contained — can be done independently.

P0 · bug fix Consolidate DB access — remove DatabaseManager singleton ~1 day

Replace all raw get_connection() + manual .close() calls with with db_session() as db:. Delete DatabaseManager or demote it to a pure connection factory. Ensures every DB operation has a deterministic commit/rollback and no connection leaks. Fix GoalTracker first as the most egregious example.

Files: anomaly_detector.py, forecaster.py, goal_tracker.py, transaction_manager.py, all *_frame.py that call get_connection() directly.

P0 · bug fix Fix module import paths in transaction_manager.py and settings_frame.py <1 hour

Change from app.repositories import → from repositories import (or set up proper __init__.py re-exports). Fix theme_engine.mode() → theme_engine.mode in SettingsFrame. These are silent runtime errors that will crash on first use.

P1 · quality Introduce a shared ui_styles.py module ~2 hours

Extract btn_primary(), btn_normal(), btn_danger(), combo_style(), input_style() into one module. All frames import from it. Reduces ~300 lines of duplication and makes a single-point style change affect the whole app. Long term: move all per-widget overrides into ThemeEngine QSS so frames have zero hardcoded style strings.

P1 · quality Add worker guard in ForecastFrame — prevent duplicate runs <1 hour

Before spawning a new ForecastWorker or AnomalyWorker, check if hasattr(self, 'fc_worker') and self.fc_worker.isRunning(): return. Same pattern already used in ChatbotFrame — apply consistently. Prevents result interleaving and potential double-writes to ai_predictions.

P1 · feature Replace hard-coded account_id=1 with account selector ~3 hours

Add a QComboBox populated from AccountRepo.get_all() to TransactionDialog. Pass selected account_id through get_data(). Update CsvImporter.import_file() to accept an account_id arg surfaced in the import dialog. This unblocks the multi-account architecture already in the schema.

P2 · infra Add physical package structure matching the logical design ~1 day (with import fixes)

Move files into app/ui/, app/core/, app/ai/, app/data/ directories with proper __init__.py. Update all import statements. This is cosmetic but matters for onboarding, IDE navigation, and preventing the accidental re-flattening that already happened (e.g. base_plugin.py is in root but imported as app.base_plugin from chatbot_frame.py).

P2 · quality Add a minimal test suite — pytest + pytest-qt ~1 week initial coverage

Priority test targets: TransactionRepo CRUD (use in-memory SQLite), TransactionModel.validate(), CsvImporter._parse_amount() and _parse_date() (pure functions, trivial to test), nlp_parser.parse_quick_add(), TransactionClassifier.predict_category_name(). These cover the highest-risk data path with minimal setup cost.

P3 · feature Complete Family/Group features ~1 week

The schema is fully designed. Needed: wire FamilyFrame to family_groups and group_members tables, implement invite-code generation + validation, filter transaction queries by owner_username (column already migrated), emit and consume the existing bus.user_switched signal to reload data when switching group member view.

Security analysis

good
Passwords stored with SHA-256 + random salt
auth_manager.py
secrets.token_hex(16) generates a cryptographically random 32-char salt per user. Hash is sha256(salt + password). Plaintext is never written to disk. Salt is unique per user so rainbow tables don't apply.
good
API keys never hardcoded — read from env / .env file
gemini_worker.py, settings_manager.py
os.getenv("GEMINI_API_KEY") with explicit empty-string check. Keys stored in a local .env file written by SettingsFrame, loaded at read-time. Not committed to source.
medium
SHA-256 is not a password-hashing function
auth_manager.py — _hash_password()
SHA-256 is a fast general-purpose hash — a GPU can compute billions per second. For passwords, use bcrypt, argon2-cffi, or Python's built-in hashlib.scrypt(). For a local desktop app the risk is low (attacker needs filesystem access), but worth fixing if the DB is ever synced to the cloud.
medium
Session file is plain JSON — trivially spoofable
session.json, auth_manager.py
{"username": "admin"} is written and read without any integrity check or expiry. An attacker with filesystem access can edit it to impersonate any user. For a local app this is acceptable, but the session should be treated as untrusted input and the username re-validated against the DB on use.
low
SQL injection risk is low — parameterised queries used throughout
repositories.py, all DB access
All SQL uses ? placeholders with tuple params. No string-interpolated SQL found. PRAGMA foreign_keys = ON is set on every connection.
low
Cloud sync sends raw SQLite binary — contains all user data
sync_manager.py
The entire finance.db is uploaded to Supabase Storage unencrypted. If the Supabase bucket is misconfigured as public, all financial data is exposed. At minimum, the bucket policy should be private + RLS. Encrypting the DB file before upload (e.g. with cryptography) would add defence in depth.

Recommended first test file

Start with tests/test_pure.py covering all "Excellent" rows above — no fixtures needed, no Qt, no DB. These 20–30 test cases will catch regressions in the most frequently used parsing and validation logic and take under an hour to write.

Overall assessment

What this codebase does really well

  • The AI plugin architecture is the standout design decision — a Strategy + Registry pattern that makes adding a 4th LLM backend a self-contained 30-line change with zero modifications to the chat UI.
  • The EventBus is properly used for decoupling. Frames don't know about each other and there are zero circular imports at the logical level.
  • The Repository pattern is clean — all SQL lives in one file, all domain objects have validation methods, and the DBContext gives atomic guarantees.
  • The ML pipeline is surprisingly complete for a student project: IsoForest anomaly detection with 5 engineered features, Prophet time-series forecasting with confidence intervals, and a full fine-tuning pipeline for DistilGPT2.
  • The UX polish is above average — toast notifications, command palette, live theming, keyboard shortcuts — features rarely seen in PyQt6 apps of this scope.

Three highest-leverage improvements

1
Fix the DB access split (1 day)
Consolidate on db_session(), remove the singleton's shared connection. This eliminates the highest-risk crash vector and the thread-safety ambiguity in one move.
2
Add a test suite (1 week)
Start with pure-function tests (parser, validator, CSV amount/date parsing). The codebase's design already makes these trivially testable — the investment is almost entirely in writing the tests, not refactoring to enable them.
3
Physical package structure (1 day)
Move files into the app/ui/, app/core/, app/ai/, app/data/ directories. Fixes the import path mismatch that causes runtime crashes and makes IDE navigation match the mental model.
